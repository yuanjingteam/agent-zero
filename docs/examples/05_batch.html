05 - 批处理：并行处理与性能优化

难度：中级 时长：~20 分钟 关键概念：并行执行、GPU 批处理、性能优化

概述

本示例演示了使用独立上下文序列（Context Sequences）对多个 LLM 请求进行并发执行，这是构建可扩展 AI 代理系统的关键技术。

性能问题

顺序处理（慢）

请求 1 ────────→ 回复 1（2 秒） ↓ 请求 2 ────────→ 回复 2（2 秒） ↓ 总计：4 秒

并行处理（快）

请求 1 ────────→ 回复 1（2 秒）──┐ ├→ 总计：2 秒 请求 2 ────────→ 回复 2（2 秒）──┘ （两个请求同时运行）

性能提升：2 倍加速！

核心概念：上下文序列

单序列 vs. 多序列

┌────────────────────────────────────────────────┐ │ 模型（只加载一次） │ ├────────────────────────────────────────────────┤ │ 上下文 │ │ ┌──────────────┐ ┌──────────────┐ │ │ │ 序列 1 │ │ 序列 2 │ │ │ │ │ │ │ │ │ │ 对话历史 A │ │ 对话历史 B │ │ │ └──────────────┘ └──────────────┘ │ └────────────────────────────────────────────────┘

关键洞察：模型权重是共享的（内存高效），每个序列拥有独立的历史记录，序列可以并行处理。

并行处理的工作原理

Promise.all 模式

顺序执行： ──────────────────────────────────── await fn1(); // 等待 2 秒 await fn2(); // 再等 2 秒 总计：4 秒 并行执行： ──────────────────────────────────── await Promise.all([ fn1(), // 立即开始 fn2() // 立即开始（不等待！） ]); 总计：2 秒（取决于最慢的那个）

执行时间线

时间 → 0 秒 1 秒 2 秒 3 秒 4 秒 │ │ │ │ │ 序列 1：├───────处理中───────┤ │ └─ 回复 1 │ 序列 2：├───────处理中───────┤ └─ 回复 2 两者都在约 2 秒完成，而不是 4 秒！

GPU 批处理

为什么批处理很重要

无批处理（低效） ────────────────────────────── GPU：[Token 1] ... 等待 ... GPU：[Token 2] ... 等待 ... GPU：[Token 3] ... 等待 ... └─ GPU 利用率不足 有批处理（高效） ───────────────────────── GPU：[Tokens 1-1024] ← 完整批次 └─ GPU 充分利用！

权衡取舍

小批次（如 128）大批次（如 2048）
内存占用更低更高
灵活性更灵活较低
吞吐量较慢更快
GPU 利用率不足充分

最佳平衡点：消费级 GPU 通常为 512-1024。

架构模式

模式 1：多用户服务

┌─────────┐ ┌─────────┐ ┌─────────┐ │ 用户 A │ │ 用户 B │ │ 用户 C │ └────┬────┘ └────┬────┘ └────┬────┘ └────────────┼────────────┘ ↓ ┌────────────────┐ │ 负载均衡器 │ └────────────────┘ ↓ ┌────────────┼────────────┐ ↓ ↓ ↓ ┌─────────┐ ┌─────────┐ ┌─────────┐ │ 序列 1 │ │ 序列 2 │ │ 序列 3 │ └─────────┘ └─────────┘ └─────────┘ └────────────┼────────────┘ ↓ ┌────────────────┐ │ 共享模型 │ └────────────────┘

模式 2：多代理系统

┌──────────────┐ │ 任务 │ └──────┬───────┘ │ ┌────────┼────────┐ ↓ ↓ ↓ ┌────────┐ ┌──────┐ ┌──────────┐ │规划代理 │ │评审代理│ │ 执行代理 │ └───┬────┘ └──┬───┘ └────┬─────┘ └─────────┼──────────┘ ↓ （全部并行运行）

资源管理

内存分配

┌──────────────────────────────────┐ │ 总显存：8GB │ ├──────────────────────────────────┤ │ 模型权重： 4.0 GB │ │ 上下文基础： 1.0 GB │ │ 序列 1（KV 缓存）： 0.8 GB │ │ 序列 2（KV 缓存）： 0.8 GB │ │ 序列 3（KV 缓存）： 0.8 GB │ │ 开销： 0.6 GB │ ├──────────────────────────────────┤ │ 总使用量： 8.0 GB │ │ 剩余： 0.0 GB │ └──────────────────────────────────┘
内存公式 所需显存 = 模型 + 上下文 + (序列数 x KV缓存)

实际场景

💬

聊天机器人服务

100 个用户：顺序处理 200 秒 vs 并行处理（10 序列）20 秒，10 倍加速。

📊

批量分析

1000 份文档：顺序 50 分钟 vs 并行（8 序列）6.25 分钟，8 倍加速。

🤖

多代理协作

规划者、分析者、执行者协同工作，快速决策而非依次等待。

限制与注意事项

  • 上下文容量共享：序列越多 = 每个序列的历史越少
  • CPU 与 GPU 并行：GPU 实现真正的并行处理，CPU 交替处理
  • 并非总是更快：仅对独立请求有帮助，对依赖请求无帮助

实用公式

加速比 加速比 = min(序列数量, 可用显存 / 每个序列的内存, GPU 计算限制)

通常：设计良好的系统可实现 2-10 倍加速。
