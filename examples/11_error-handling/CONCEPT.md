## 概念：智能体的全面错误处理

智能体比普通应用更容易失败，因为它们编排了**多个不可靠的步骤**：

- **LLM 调用**（超时、资源限制、格式错误的输出、运行时异常）
- **工具执行**（网络故障、无效输入、服务不可用）
- **工作流逻辑**（策略拦截、部分完成、跨工具的依赖链）

本示例使用三个理念来使故障变得安全且可理解：

### 1) 标准化的错误分类体系

使用一小组带有**稳定错误码**和一致字段的错误类：

- **`ValidationError`**：用户输入缺失/无效（快速失败；通常不可重试）
- **`LLMCallError`**：LLM 提供商/模型调用失败或返回不可用的输出（通常可重试）
- **`ToolExecutionError`**：工具失败（有时可重试，有时不可重试）
- **`AgentWorkflowError`**：**编排级别**的失败 — 多步骤运行无法按设计完成。

在生产环境中，你可能会将 **`AgentWorkflowError`** 拆分为更细粒度的类型（策略、工作流链、全系统中断）。本课程使用**一个类**配合 **`step`** 字段和简短的源码注释，使**相同的结构**可以在小型演示中代表这些概念。

本仓库中的示例：

- **`policy_guard`** — 验证之后，拦截器阻止了请求（演示：用户提到 **`u_demo_workflow`**）。
- **`resolve_user_profile`** — 在降级模式下，**主工具重试耗尽**，**尝试备用工具**，**备用工具也失败**；上报的错误是工作流级别的，内部工具错误作为 **`cause`** 保留。

每个错误包含：

- **`code`**：机器可读的稳定标识符（适合指标和告警）
- **`userMessage`**：展示给用户的安全、非技术性消息
- **`retryable`**：是否适合自动重试
- **`details`**：用于日志的结构化数据（工具名、步骤名、ID 等）
- **`cause`**：原始错误（保留根本原因链）

**`AgentWorkflowError`** 额外携带 **`step`**。其 `details` 会将 `step` 与你传入的任何额外元数据合并。

### 2) 分类与恢复策略

**先标准化，再分类。** **`normalizeUnknownError`** 将任意抛出的值转换为 **`AppError`**。**`classifyError`** 添加 **`retryable`** 和 **`type`**（`error.code`），使重试、日志和用户消息共享同一管道，而不是重复 `instanceof` 判断树。

恢复策略（典型的阶梯式），如通过 `node-llama-cpp` 使用真实本地 LLM 所示：

- **超时**：限制任何步骤的停滞时间
- **重试**：仅当 **`classifyError`** 判定为 **`retryable`** 时（带退避和抖动）
- **备用方案**：如果主工具以**瞬态**方式失败，运行更安全的替代方案，返回降级但有用的结果
- **降级模式**：如果 LLM 路径失败，委托给 **`runDegradedProfileResolution`** — 确定性地提取 **`u_<digits>`** ID 并使用相同的**主工具 > 备用工具**模型，而不是将该逻辑内联在一个巨大的 `catch` 中
- **优雅失败**：如果无法恢复，返回 **`formatUserFacingError`** 加上关联 ID

当降级模式下**主工具**（重试后）和**备用工具都**失败时，本示例将结果提升为 **`AgentWorkflowError`**：用户仍然看到一条清晰的消息，而 **`cause`** 保留了底层工具故障信息以供调试。

### 3) 将用户消息与调试信息分离

用户应该看到：

- 清晰的下一步操作（重试、重新表述、缩短输入）
- 没有堆栈跟踪或提供商内部信息
- 一个可以提供给客服的**引用 ID**

开发者/运维人员应该看到：

- 稳定的错误 `code`
- 结构化的 `details`
- 关联 ID
- 原始 **`cause`**

对于 **`AgentWorkflowError`**，本示例还会打印一个**控制台横幅**（步骤、错误码、关联 ID、消息、详情、cause 摘要），使实时演示和本地调试在紧凑的 `[agent_error]` 日志行旁边依然可读。

### 确定性演示与 `SIMULATION`

为了保持教学运行的可预测性，用户 ID **`u_999`** 和 **`u_777`** 由一个小的 **`SIMULATION`** 映射驱动（参见 `error-handling.js`）：

- 文本中的 **`u_demo_workflow`** 触发 **`policy_guard`**。
- **`SKIP_LLM_DEGRADED u_777`** 跳过 LLM，进入降级模式，并使用 **`u_777`** 使主工具和备用工具都以可复现的方式失败。

### 为什么这种模式可扩展

- **一致性**：所有故障都以相同方式呈现；未知错误在处理前被标准化
- **可观测性**：指标/告警按 `code` 和工作流 `step` 分组
- **安全性**：敏感/提供商特定的细节不会出现在用户消息中
- **弹性**：瞬态问题自动恢复；硬故障降级或作为单个工作流级别错误上报，保留 **`cause`**
