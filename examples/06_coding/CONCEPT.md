# 概念：流式传输与响应控制

## 概述

本示例演示了**流式响应（Streaming Responses）**和 **token 限制（Token Limits）**，这是构建具有受控输出的响应式 AI 智能体的两项核心技术。

## 流式传输问题

### 传统（非流式）方式

```
用户发送提示词
       ↓
   [等待 10 秒...]
       ↓
完整响应一次性出现
```

**问题：**
- 用户体验差（长时间等待）
- 无进度指示
- 无法中断错误响应
- 感觉响应迟钝

### 流式方式（本示例）

```
用户发送提示词
       ↓
"Hoisting" (0.1 秒) → 用户看到第一个词！
       ↓
"is a" (0.2 秒) → 更多文本出现
       ↓
"JavaScript" (0.3 秒) → 持续反馈
       ↓
[逐 token 继续...]
```

**优势：**
- 即时反馈
- 进度可见
- 可提前中断
- 感觉互动性强

## 流式工作原理

### 逐 Token 生成

LLM 在内部一次生成一个 token。流式传输将这一过程暴露出来：

```
LLM 内部过程：
┌─────────────────────────────────────┐
│  Token 1: "Hoisting"                │
│  Token 2: "is"                      │
│  Token 3: "a"                       │
│  Token 4: "JavaScript"              │
│  Token 5: "mechanism"               │
│  ...                                │
└─────────────────────────────────────┘

无流式：                有流式：
等待所有 token            立即发出每个 token
└─→ 缓冲 → 返回        └─→ 回调 → 显示
```

### onTextChunk 回调

```
┌────────────────────────────────────┐
│        模型生成                     │
└────────────┬───────────────────────┘
             │
    ┌────────┴─────────┐
    │  每个新 token     │
    └────────┬─────────┘
             ↓
    ┌────────────────────┐
    │ onTextChunk(text)  │  ← 你的回调
    └────────┬───────────┘
             ↓
    你的代码处理它：
    • 显示给用户
    • 通过网络发送
    • 记录到文件
    • 分析内容
```

## Token 限制：maxTokens

### 为什么要限制输出？

没有限制时，模型可能生成：
```
用户："解释一下变量提升"
模型：[生成 10,000 个单词，包括：
       - 完整的 JavaScript 历史
       - 所有边界情况
       - 不相关的示例
       - 永不停止...]
```

有限制时：
```
用户："解释一下变量提升"
模型：[生成约 1500 个单词
       - 核心概念
       - 关键示例
       - 在 2000 个 token 处停止]
```

### Token 预算

```
上下文窗口：4096 个 token
├─ 系统提示词：200 个 token
├─ 用户消息：100 个 token
├─ 响应（maxTokens）：2000 个 token
└─ 剩余用于历史记录：1796 个 token

总使用量：2300 个 token
可用量：1796 个 token 用于未来对话
```

### 成本与质量

```
Token 限制         输出质量            使用场景
───────────       ───────────────     ─────────────────
100               简短，可能被截断    快速回答
500               简洁但完整          简短解释
2000（示例）      详细                完整解释
无限制            有冗长风险          长度未知时
```

## 实时应用场景

### 模式 1：交互式 CLI

```
用户："解释闭包"
       ↓
终端："A closure is a function..."
      （逐字出现，像打字一样）
       ↓
用户看到进度，知道正在处理
```

### 模式 2：Web 应用

```
浏览器                    服务器
   │                         │
   ├─── 发送提示词 ─────────→│
   │                         │
   │←── 数据块 1: "Closures"─┤
   │    （立即显示）           │
   │                         │
   │←── 数据块 2: "are"──────┤
   │    （追加显示）           │
   │                         │
   │←── 数据块 3: "functions"─┤
   │    （继续追加...）        │
```

实现方式：
- 服务器推送事件（SSE）
- WebSocket
- HTTP 流式传输

### 模式 3：多消费者

```
         onTextChunk(text)
                │
        ┌───────┼───────┐
        ↓       ↓       ↓
    控制台    WebSocket  日志文件
    显示    → 客户端    → 存储
```

## 性能特征

### 延迟与吞吐量

```
首个 Token 时间（TTFT）：
├─ 小型模型（1.7B）：约 100ms
├─ 中型模型（8B）：约 200ms
└─ 大型模型（20B）：约 500ms

每秒 Token 数：
├─ 小型模型：50-80 tok/s
├─ 中型模型：20-35 tok/s
└─ 大型模型：10-15 tok/s

用户体验：
TTFT < 500ms → 感觉即时
Tok/s > 20 → 阅读自然
```

### 资源权衡

```
模型大小      内存      速度      质量
──────────     ────────   ─────     ───────
1.7B           约 2GB     快        良好
8B             约 6GB     中等      更好
20B            约 12GB    较慢      最佳
```

## 高级概念

### 缓冲策略

**无缓冲（即时）**
```
每个 token → 回调 → 显示
└─ 最流畅的用户体验，但开销更大
```

**行缓冲**
```
累积到换行符 → 刷新
└─ 更适合基于段落的输出
```

**时间缓冲**
```
累积 50ms → 批量刷新
└─ 降低回调频率
```

### 提前停止

```
生成进行中：
"The answer is clearly... wait, actually..."
                         ↑
                  onTextChunk 检测到问题
                         ↓
                   停止生成
                         ↓
              "Let me reconsider"
```

适用于：
- 检测偏离主题的响应
- 安全过滤
- 相关性检查

### 渐进增强

```
部分响应分析：
┌─────────────────────────────────┐
│ "To implement this feature..."  │
│                                 │
│ ← 已经是有用的信息               │
│                                 │
│ "...you'll need: 1) Node.js"    │
│                                 │
│ ← 可以开始据此行动               │
│                                 │
│ "2) Express framework"          │
└─────────────────────────────────┘

智能体可以在响应完成前就开始工作！
```

## 上下文大小感知

### 为什么重要

```
┌────────────────────────────────┐
│    上下文窗口 (4096)            │
├────────────────────────────────┤
│ 系统提示词          200 token  │
│ 对话历史            1000       │
│ 当前提示词          100        │
│ 响应空间            2796       │
└────────────────────────────────┘

如果 maxTokens > 2796：
└─→ 错误或截断！
```

### 动态调整

```
可用量 = contextSize - (prompt + history)

if (maxTokens > available) {
    maxTokens = available;
    // 或清除旧历史记录
}
```

## 智能体架构中的流式传输

### 简单智能体

```
用户 → LLM（流式） → 显示
       └─ onTextChunk 显示进度
```

### 多步骤智能体

```
步骤 1：规划（流式）→ 显示思考过程
步骤 2：行动（流式）→ 显示操作
步骤 3：结果（流式）→ 显示结果
       └─ 用户看到智能体的全过程
```

### 协作式智能体

```
智能体 A（流式）──┐
                  ├─→ 协调器 → 用户
智能体 B（流式）──┘
       └─ 两者同时流式传输
```

## 最佳实践

### 1. 始终设置 maxTokens

```
✓ 推荐：
session.prompt(query, { maxTokens: 2000 })

✗ 有风险：
session.prompt(query)
└─ 可能使用整个上下文！
```

### 2. 处理部分更新

```
let fullResponse = '';
onTextChunk: (chunk) => {
    fullResponse += chunk;
    display(chunk);        // 立即显示
    logComplete = false;   // 标记未完成
}
// 完成后：
saveToDatabase(fullResponse);
```

### 3. 提供反馈

```
onTextChunk: (chunk) => {
    if (firstChunk) {
        showLoadingDone();
        firstChunk = false;
    }
    appendToDisplay(chunk);
}
```

### 4. 监控性能

```
const startTime = Date.now();
let tokenCount = 0;

onTextChunk: (chunk) => {
    tokenCount += estimateTokens(chunk);
    const elapsed = (Date.now() - startTime) / 1000;
    const tokensPerSecond = tokenCount / elapsed;
    updateMetrics(tokensPerSecond);
}
```

## 关键要点

1. **流式传输改善用户体验**：用户立即看到进度
2. **maxTokens 控制成本**：防止无限制生成
3. **逐 token 生成**：LLM 一次生成一个 token
4. **onTextChunk 回调**：你接入生成过程的钩子
5. **上下文感知很重要**：监控可用空间
6. **生产环境必备**：实时系统需要流式传输
